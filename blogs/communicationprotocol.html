Communication Protocols in Industry — UART, I2C, SPI, LIN, RS485 & MODBUS Explained

A practical deep dive into the most commonly used communication protocols in embedded and industrial systems — when to use each, their trade-offs, pin configurations, and real code examples. Based on real-world experience using these protocols in automotive, industrial automation, and IoT projects.

UARTI2CSPI RS485MODBUSLINCAN Bus

Protocol Overview

In my 10+ years of embedded systems work — from crane SLI systems to automotive AFCS — I've used all of these protocols in production hardware. The choice of protocol can make or break your design: wrong choice means rework, noise issues, or system instability. This article is the guide I wish I had when starting out.

💡 Key insight: There is no "best" protocol. The right choice depends on distance, speed, number of devices, wiring constraints, and noise environment. Understanding all of them gives you the flexibility to design the right system.

UART — Universal Asynchronous Receiver-Transmitter
Simple, ubiquitous, the default for debug output

UART

UART is the simplest serial protocol — asynchronous, point-to-point, and used everywhere from GPS modules to PC debug consoles. It uses just two wires: TX and RX.

Key Parameters

  • Baud Rate: Common rates — 9600, 115200, 1Mbps. Both devices must match exactly.
  • Frame: 1 start bit + 8 data bits + optional parity + 1–2 stop bits
  • Flow Control: Hardware (RTS/CTS) or software (XON/XOFF)

Arduino Example

void setup() {
  Serial.begin(115200);   // Set baud rate
}
void loop() {
  Serial.println("Hello from Sandeep!");
  delay(1000);
}

When to Use UART

  • Debug output during development (most common use)
  • GPS, GSM, Bluetooth modules (HC-05, SIM800)
  • PC ↔ Microcontroller communication
  • When you only need two devices talking to each other
I2C — Inter-Integrated Circuit
Multi-device on 2 wires — ideal for sensors and displays

I2C

I2C is a synchronous, multi-master, multi-slave protocol using just two wires — SDA (data) and SCL (clock). Every device has a unique 7-bit address, allowing up to 127 devices on one bus.

Key Parameters

  • Speed: Standard 100kbps / Fast 400kbps / Fast+ 1Mbps / HS 3.4Mbps
  • Addressing: 7-bit (127 devices) or 10-bit (1023 devices)
  • Pull-ups required: SDA and SCL need pull-up resistors (typically 4.7kΩ)

Reading a Temperature Sensor (LM75) on Arduino

#include <Wire.h>

void setup() {
  Wire.begin();           // Join I2C bus as master
  Serial.begin(115200);
}
void loop() {
  Wire.requestFrom(0x48, 2);   // LM75 default address
  if (Wire.available() >= 2) {
    int msb = Wire.read();
    int lsb = Wire.read();
    float temp = ((msb << 8) | lsb) >> 5;
    Serial.print("Temp: ");
    Serial.println(temp * 0.125);
  }
  delay(1000);
}

⚠️ Gotcha: Missing or incorrect pull-up resistors is the #1 cause of I2C communication failure. If your bus hangs, check your pull-ups first.

SPI — Serial Peripheral Interface
Fast, full-duplex — for high-speed peripherals

SPI

SPI is a synchronous, full-duplex protocol with 4 wires: MOSI, MISO, SCK, and CS (Chip Select). Fastest of the common protocols — routinely used at 10–80MHz for flash memory, displays, and ADCs.

Key Parameters

  • CPOL & CPHA: 4 modes define clock polarity and phase. Match the peripheral's datasheet.
  • CS per device: Each slave needs its own Chip Select line.
  • No addressing: CS line selects the active device instead.

When to Use SPI

  • Flash/EEPROM memory, SD cards
  • TFT/OLED displays
  • High-speed ADC/DAC
  • When speed matters more than pin count
LIN — Local Interconnect Network
Single-wire, low-cost automotive protocol

LIN

LIN is a low-cost, single-wire serial protocol primarily used in automotive applications for lower-priority subsystems — seat adjustments, window motors, lighting, HVAC. It runs at 20kbps maximum.

Frame Structure

Sync Break | Sync Byte (0x55) | Identifier | Data (1-8 bytes) | Checksum

LIN vs CAN

  • LIN: Single master, 1 wire, up to 20kbps, <16 nodes — cheap, simple
  • CAN: Multi-master, 2 wires, up to 1Mbps, multi-node — complex, reliable, safety-critical
RS485 + MODBUS
Industrial workhorse — long distance, noise-immune

RS485 & MODBUS

RS485 uses differential signalling on two wires (A and B) — making it immune to common-mode noise and capable of cable runs up to 1200 metres at up to 10Mbps. I used RS485 extensively in the SLI crane project for sensor-to-controller communication over long cable runs in noisy industrial environments.

MODBUS RTU runs on top of RS485 as the application protocol — it's the universal language of industrial automation.

MODBUS Frame

Slave Address | Function Code | Data | CRC16

Arduino MODBUS Example

#include <ModbusRtu.h>

Modbus master(0, Serial, 0);  // 0 = master, Serial port, RE/DE pin

void setup() {
  Serial.begin(9600);
  master.begin(9600);
}
void loop() {
  master.poll();  // Process MODBUS requests
}

When to Use RS485 + MODBUS

  • Industrial automation (PLCs, sensors, actuators)
  • Long cable runs (>15m) in noisy environments
  • Multi-drop networks (up to 32 devices on one bus)
  • Energy meters, building automation, HVAC

When to Use What — Decision Guide

  • 📍 Short distance, 1 device: UART
  • 📍 Multiple sensors on same board: I2C
  • 📍 Need speed (display, memory): SPI
  • 📍 Automotive body control, low cost: LIN
  • 📍 Industrial, long cables, noisy environment: RS485 + MODBUS
  • 📍 Safety-critical automotive, real-time: CAN Bus

💡 Pro tip from the field: In my SLI crane system, we used RS485 for the long cable run from sensor nodes to the central controller, and I2C internally on each PCB for sensor arrays. The right tool for the right job.
