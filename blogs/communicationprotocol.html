📡 Communication Protocols

Communication Protocols — UART, I2C, SPI, LIN, RS485 & CAN Explained

A practical deep dive with animated signal diagrams, real code examples, and field-tested tips. Based on 10+ years using these protocols in automotive, industrial automation, IoT, and avionics projects.

Protocol Overview

In 10+ years of embedded systems — from crane SLI systems to automotive AFCS — I've used every protocol covered here in production hardware. The wrong choice means rework, noise issues, or system instability. This article is the guide I wish I had at the start.

💡 Key insight: There is no "best" protocol. The right choice depends on distance, speed, number of devices, wiring constraints, and noise environment. Understanding all of them gives you the design flexibility to build the right system.

Fig 1 — Protocol comparison at a glance. Each row animates in. RS485 uniquely supports 1200m cable runs — essential for industrial sites like crane installations.
🔗
UART — Universal Asynchronous Receiver-Transmitter
Simple, ubiquitous, the default for debug output and module communication
2 wires Asynchronous Point-to-point only up to 1Mbps

UART

UART is the simplest serial protocol — asynchronous (no shared clock), point-to-point, and used everywhere from GPS modules to PC debug consoles. It uses just two wires: TX (transmit) and RX (receive). Crucially, because it's asynchronous, both devices must agree on the same baud rate.

Fig 2 — UART frame animation. One start bit (low) + 8 data bits (LSB first) + one stop bit (high) = 10 bit times per byte. At 115200 baud each bit is 8.68µs. This frame transmits 'A' (0x41 = 01000001).

Key Parameters

  • Baud Rate: Both devices must match exactly — 9600, 115200, 1Mbps. Mismatch = garbage data.
  • Frame format (8N1): 8 data bits, No parity, 1 stop bit — the most common configuration.
  • Flow Control: Hardware (RTS/CTS lines) or software (XON/XOFF characters)
  • Voltage levels: TTL (0/3.3V or 0/5V) for MCU pins. RS-232 uses ±12V — need a level shifter.

Hardware Flow Control — RTS & CTS

Without flow control, a fast transmitter can overwhelm a slow receiver's buffer — causing dropped bytes with no error flag. RTS (Request To Send) and CTS (Clear To Send) are two extra wires that let both devices negotiate when it is safe to transmit. This is called hardware flow control.

💡 Analogy: Think of RTS/CTS like traffic lights at a narrow bridge. The receiver controls the light (CTS). The transmitter waits at red (CTS HIGH = stop) and only crosses when the light turns green (CTS LOW = go). No data is ever pushed onto an overwhelmed receiver.

RTS — Request To Send
Output pin on the transmitter. Driven LOW to say "I have data — are you ready?". Driven HIGH when the transmitter has nothing to send or is pausing. Connected to the receiver's CTS pin.
Output on TX device LOW = "ready to send"
CTS — Clear To Send
Input pin on the transmitter. Driven LOW by the receiver to say "buffer ready — send now". Goes HIGH when the receiver's buffer is full — forcing the transmitter to pause. Connected to the transmitter's RTS pin on the other end.
Input on TX device LOW = "safe to send"

⚠️ Cross-wiring rule: RTS of Device A → CTS of Device B. RTS of Device B → CTS of Device A. This is the single most common wiring mistake. On a straight-through cable both pins short to each other — you need a null-modem cable or cross-over jumpers.

Fig 2b — 4-wire UART connection with hardware flow control. TX→RX carry data (straight across). RTS of A crosses to CTS of B, and RTS of B crosses to CTS of A. The indicator circles show RTS asserted (green = LOW = "I want to send") and CTS acknowledged (teal = LOW = "safe to send"). Animated packets flow only when CTS is LOW.
Fig 2c — Full RTS/CTS handshake timing. Five phases: ① both idle, ② A asserts RTS to request permission, ③ B grants CTS and data flows (H-e-l-l), ④ B's buffer fills so it raises CTS HIGH — A freezes mid-message without losing data, ⑤ buffer drains, B re-asserts CTS LOW, A resumes. The RX buffer bar at the bottom shows fill level turning red at capacity.

When Do You Actually Need RTS/CTS?

✅ Use RTS/CTS when…
  • High baud rate (>115200) + slow receiver
  • Large bursts of data (file transfer, firmware OTA)
  • Receiver has small FIFO buffer (<16 bytes)
  • GPS modules at 1Mbps baud
  • Bluetooth HC-05 modules in bulk transfer
❌ Skip RTS/CTS when…
  • Only 2 wires available (TX, RX)
  • Low baud rate (9600–115200) with small packets
  • Debug console / simple sensor readout
  • Software handles retries / framing anyway
  • MCU pins are scarce

Code Example — MCU UART Init

// UART setup — transmit "Hello" at 115200 baud
void setup() {
  Serial.begin(115200);     // Set baud rate — must match receiver
}

void loop() {
  Serial.println("Hello from Sandeep!");
  delay(1000);

  // Read incoming data
  if (Serial.available() > 0) {
    char c = Serial.read();
    Serial.print("Received: ");
    Serial.println(c);
  }
}
🔄
I2C — Inter-Integrated Circuit
Multi-device on just 2 wires — the go-to for sensors and displays on the same PCB
2 wires (SDA/SCL) Synchronous 127 devices 100k–3.4Mbps

I2C

I2C uses two bidirectional open-drain wires — SDA (data) and SCL (clock). The master generates the clock; every device has a unique 7-bit address, allowing up to 127 nodes on a single pair of wires. Both lines require pull-up resistors to VCC — typically 4.7kΩ for standard mode.

Fig 3 — I2C multi-device bus. One master (MCU) communicates with a temperature sensor (0x48), OLED display (0x3C), and EEPROM (0x50) over the same two wires. Animated dots show address packets being broadcast — only the addressed slave responds.

Key Parameters

  • Speed modes: Standard 100kbps / Fast 400kbps / Fast+ 1Mbps / High-Speed 3.4Mbps
  • Addressing: 7-bit (127 devices) or 10-bit (1023 devices)
  • Pull-ups required: SDA and SCL are open-drain — need 4.7kΩ resistors to VCC. This is the #1 setup mistake.
  • ACK/NAK: Every byte is acknowledged. If a device is absent, the master gets a NAK — no ghost reads.

Code Example — Reading LM75 Temperature Sensor

#include <Wire.h>

void setup() {
  Wire.begin();              // Join I2C bus as master
  Serial.begin(115200);
}

void loop() {
  Wire.beginTransmission(0x48);  // Address LM75 (pins A0,A1,A2 = GND)
  Wire.write(0x00);              // Point to temperature register
  Wire.endTransmission(false);   // Repeated START — keep bus active

  Wire.requestFrom(0x48, 2);     // Read 2 bytes MSB + LSB
  if (Wire.available() >= 2) {
    int16_t raw = (Wire.read() << 8) | Wire.read();
    float temp = raw / 256.0;    // 9-bit resolution, 0.5°C/bit
    Serial.print("Temperature: ");
    Serial.print(temp);
    Serial.println(" °C");
  }
  delay(1000);
}

⚠️ Gotcha: Missing or wrong pull-up resistors is the #1 cause of I2C failures. If your bus hangs or you get 0xFF reads, check pull-ups first. Value depends on bus capacitance — 4.7kΩ for short PCB traces, 2.2kΩ for longer runs.

⚡
SPI — Serial Peripheral Interface
Full-duplex, high-speed — for flash memory, displays, and ADCs
4 wires Full-duplex up to 80Mbps No addressing

SPI

SPI is synchronous and full-duplex — MOSI and MISO carry data simultaneously in both directions while SCK clocks every bit. CS (Chip Select, active low) selects one device at a time. There's no addressing — the CS line does the selection. Fastest common protocol: routinely used at 10–80MHz.

Fig 4 — SPI timing diagram. CS goes low to select the slave. SCK clocks 8 bits simultaneously on MOSI (master→slave, teal) and MISO (slave→master, gold). Sampled on rising edge (Mode 0). Full duplex — both directions happen in the same clock cycle.

Key Parameters

  • CPOL & CPHA: 4 modes (0–3) define clock polarity and phase. Always match the peripheral's datasheet. Wrong mode = corrupt data with no error indication.
  • One CS per slave: Each additional device needs a dedicated CS line from the master.
  • No ACK, no addressing: If the slave doesn't respond, you won't know from the protocol alone — add application-level checksums for critical data.
Fig 4b — SPI multi-slave working. Three slaves share MOSI, MISO, SCK — each gets a dedicated CS line. Only the slave whose CS is pulled LOW is active; others tri-state their MISO output. Animated glowing border and data packets show which device is currently selected.
🚗
LIN — Local Interconnect Network
Single-wire, low-cost automotive protocol for body control
1 wire 20kbps max 1 master up to 16 nodes

LIN

LIN is a low-cost, single-wire serial protocol for lower-priority automotive subsystems — seat position motors, mirror adjustment, window controls, HVAC, ambient lighting. A modern car uses LIN for everything that doesn't need CAN's speed or robustness.

Fig 5 — LIN frame (top): Break field wakes all slaves → Sync byte locks baud rate → PID selects which slave → Data published → Checksum. Bottom: A typical BCM LIN cluster in a car door — seats, mirrors, windows, HVAC, ambient LED all on one wire, master polling each slave in schedule.

LIN vs CAN — When to Use Which

  • LIN: Single master, 1 wire, ≤20kbps, ≤16 nodes, ≤40m — cheap, simple, no arbitration
  • CAN: Multi-master, 2 wires, ≤1Mbps, multi-node, 500m — complex, reliable, safety-critical
  • Rule of thumb: If it's a body comfort function and 20kbps is enough → LIN. If it's powertrain or safety → CAN.
🏭
RS485 + MODBUS RTU
Industrial workhorse — 1200m range, noise-immune differential signalling
2 wires (differential) 1200m! 32 devices 10Mbps

RS485 & MODBUS

RS485 uses differential signalling — the voltage difference between A and B carries the data, not the absolute voltage. This means common-mode noise (from motors, inverters, welding equipment) cancels out. I used RS485 in my SLI crane project for sensor-to-controller runs across noisy factory floors.

Fig 5 — RS485 differential signalling. Noise spikes hit both A and B lines equally. The receiver measures A–B, so the noise cancels out. This is why RS485 can reliably communicate over 1200m in a noisy factory environment where single-ended protocols would fail.

MODBUS RTU is the application-layer protocol that runs on RS485 in industrial automation — it's the universal language of PLCs, VFDs, energy meters, and SCADA systems.

MODBUS Frame

| Slave Addr (1B) | Function Code (1B) | Data (N bytes) | CRC16 (2B) |

// Example: Read 2 holding registers from slave 0x01, starting at addr 0x0000
TX: 01 03 00 00 00 02 C4 0B
    ^^ ^^ ^^^^^ ^^^^^ ^^^^^
    |  |  |     |     CRC16
    |  |  |     Register count
    |  |  Start address
    |  Function code (0x03 = Read Holding Registers)
    Slave address
Fig 6b — RS485 MODBUS RTU multi-drop network. Master PLC/SCADA polls each slave in turn by address. Animated red packets = request (master→slave); green packets = response (slave→master). Half-duplex — only one device transmits at a time. 120Ω termination resistors prevent reflections at both cable ends.

When to Use RS485 + MODBUS

  • Industrial automation: PLCs, VFDs, sensors, actuators
  • Long cable runs (>15m) in noisy environments (motors, inverters)
  • Multi-drop networks — up to 32 devices on one twisted pair
  • Energy meters, building automation, HVAC, water treatment
🚘
CAN Bus — Controller Area Network
Multi-master, fault-tolerant — the backbone of automotive and safety-critical systems
2 wires (diff) 1Mbps Multi-master 500m

CAN Bus

CAN is a multi-master differential bus originally designed for automotive environments. Unlike I2C or SPI, any node can transmit without a master — arbitration is non-destructive using message priority IDs. Used in my AFCS project for real-time flight control data distribution.

Fig 7b — CAN frame (top) and multi-master arbitration (bottom). Two nodes start transmitting simultaneously. The node with lower ID (0x0A1) wins — it reads its own bits on the bus and sees agreement. The higher-ID node (0x1F4) detects a mismatch, backs off immediately without corrupting the frame, and retries. Animated gold packet shows the winning message propagating to all listeners.

CAN Frame Structure

| SOF | Arbitration ID (11/29 bit) | Control | Data (0–8B) | CRC | ACK | EOF |
  ^                 ^                                              ^
  Start of frame    Message priority (lower ID = higher priority)  Error detection

Key CAN Features

  • Non-destructive arbitration: Multiple nodes can start transmitting simultaneously — lower ID wins, others back off and retry. No data corruption.
  • Error detection & confinement: CRC, bit stuffing, and frame check. Faulty nodes are automatically isolated from the bus.
  • ISO 11898 standard: Used in automotive (SAE J1939, OBD-II), industrial (CANopen, DeviceNet), and medical (CAN in Automation).
  • Extended IDs: Standard 11-bit or extended 29-bit message identifiers for large networks.

When to Use What — Decision Guide

📍
Short distance, 2 devices, simple?
→ UART
📍
Multiple sensors on same PCB?
→ I2C
📍
Need maximum speed (display, flash)?
→ SPI
📍
Automotive body control, low-cost?
→ LIN
📍
Industrial, long cable, noisy site?
→ RS485 + MODBUS
📍
Safety-critical, real-time, automotive?
→ CAN Bus

💡 Field insight: In my SLI crane system, I used RS485 for the long runs from sensor nodes to the central controller, and I2C internally on each PCB for the sensor array. The right tool for the right layer of the system — not one protocol for everything.
