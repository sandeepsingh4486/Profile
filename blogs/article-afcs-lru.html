§01 — Introduction: The LRU at the Heart of Every Safe Flight

Testing an LRU is not just running software — it is proving, to mathematical certainty, that the code controlling your control surfaces will not kill anyone.

An Automated Flight Control System (AFCS) is a safety-critical avionics architecture responsible for maintaining aircraft stability and executing pilot commands. A Line Replaceable Unit (LRU) is the fundamental modular building block — a self-contained, removable avionics box designed for rapid maintenance replacement.

§02 — Avionics Bus Topology

Understanding the bus topology is essential for designing the LRU Test Bench (LTB) correctly — every interface must be simulated with the right hardware cards, at the right timing, with the right error injection capability.

ARINC 429 — Why One-Way Architecture Is a Safety Feature

ARINC 429 is strictly one transmitter → up to 20 receivers. No receiver can respond on the same wire pair. A failed receiver cannot corrupt data for other receivers. A transmitter fault affects only that label stream — it cannot take down the whole bus. This architectural simplicity is a fundamental safety benefit, not a limitation.

§05 — Hardware-in-the-Loop (HIL) Simulation

HIL places the physical LRU inside a closed-loop real-time simulation of the aircraft. The LRU believes it is flying. Every failure mode unsafe to evaluate in the air can be safely tested in the lab.

💡 Why HIL is mandatory for DAL A: DO-178C requires coverage of all executable code paths (MC/DC for DAL A). Many failure branches — sensor failure, bus timeout, redundancy switchover — are physically impossible to trigger in a real aircraft without risk. HIL makes these branches reachable, testable, and certifiable.

⚠ Label 0xC001 WARN: This is an intentionally injected fault for negative test coverage — verifying that the FCC correctly handles a system status word warning. The 0 error count on 12,844 words confirms the bus is clean under nominal conditions. This negative test case satisfies DO-178C MC/DC coverage for the warning-handling code branch.

💡 MC/DC — Modified Condition/Decision Coverage: Required at DAL A. Every condition in every decision must independently affect the decision outcome. This is the most rigorous software test coverage criterion in DO-178C — it proves that no single code decision can fail silently without a test detecting it.
